## Top 5 高优先级问题

### 1. FB-001 — 单文件架构维护瓶颈

- **问题描述**：后端 `mysql_multi_source_main.py` 3700 行，前端 `index.html` 2500 行，虽已拆出 6 个 mixin 模块，但主文件仍承载大量业务逻辑。
- **用户/开发者原话**：Review 报告指出"单文件脚本体量大，后续可进一步按模块拆分"。
- **影响分析**：新功能开发效率低，代码冲突风险高，新人上手成本大。
- **建议方案**：将剩余业务逻辑按功能域拆分（source_manager、bootstrap_manager、replication_manager），前端拆为 api/state/render/actions 多文件。

### 2. FB-004 — GTID purged 注入风险

- **问题描述**：`_auto_start_channel_after_bootstrap` 中 `captured_gtid` 直接拼入 `SET @@GLOBAL.gtid_purged = '{}'`，若 GTID 值含单引号可导致 SQL 注入。
- **代码位置**：`mysql_multi_source_main.py:2520`
- **影响分析**：虽然 GTID 正常格式不含特殊字符，但恶意构造的 task 数据可利用此点。
- **建议方案**：对 `captured_gtid` 做正则校验（`^\d+:\d+(-\d+:\d+)*(,\d+:\d+(-\d+:\d+)*)*$`），不匹配则拒绝执行。

### 3. FB-007 — 配置单明文密码泄露

- **问题描述**：`master_export_signed_profile` 将 `repl_password` 明文放入 payload 并 base64 编码输出，任何获取配置单的人可直接读取密码。
- **代码位置**：`mms/handshake_service.py:23`
- **影响分析**：配置单可能通过聊天工具传输，泄露面广。
- **建议方案**：配置单中密码字段使用 Fernet 加密，导入时解密；或改为临时凭证 + 一次性 token 模式。

### 4. FB-008 — MySQL 8.0.23+ 语法兼容

- **问题描述**：全部复制命令使用 `CHANGE MASTER TO` / `START SLAVE` / `STOP SLAVE`，MySQL 8.0.23 已标记废弃，8.4+ 可能移除。
- **社区反馈**：宝塔用户普遍使用 MySQL 8.0，升级到 8.4 后将出现语法错误。
- **建议方案**：运行时检测 MySQL 版本，8.0.23+ 使用 `CHANGE REPLICATION SOURCE TO` / `START REPLICA` / `STOP REPLICA`。

### 5. FB-011 — 宝塔 my.cnf 覆盖风险

- **问题描述**：宝塔面板操作 MySQL（重启、升级、配置修改）时可能覆盖 `/etc/my.cnf`，导致多源复制配置丢失。
- **社区反馈**：这是宝塔用户配置 MySQL 复制时最常遇到的问题之一。
- **建议方案**：将多源复制配置写入 `/etc/my.cnf.d/multi_source.cnf`（include 方式），或在宝塔 MySQL 配置中注入 include 指令。
