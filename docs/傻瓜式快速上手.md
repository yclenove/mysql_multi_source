# 傻瓜式快速上手（小白专用）

这份文档目标只有一个：  
**让你不懂主从原理，也能把插件配置成功并跑起来。**

本篇只讲一件事：**主库步骤在哪做、从库步骤在哪做**。  
先强调一个关键事实：插件里的 `主库检测/主库修复/创建复制账号/重启MySQL` 都作用在**插件所在机器本地 MySQL**。  
所以：
- 要操作主库配置，请在**主库服务器的宝塔面板**打开插件执行；
- 要配置复制来源、映射、初始化任务，请在**从库服务器的宝塔面板**执行。

---

## 一、先准备 6 个信息

写在记事本里，后面会反复用到：

1. 从库 IP（例如 `10.0.0.2`）
2. 主库 IP（例如 `10.0.0.11`）
3. 主库端口（一般 `3306`）
4. 复制账号（例如 `repl_user`）
5. 复制密码（例如 `Repl@2026!`）
6. 主库库名与从库目标库名（例如 `order_db -> m1_order_db`）

---

## 二、在哪台机器操作（先看这张表）

| 你在做什么 | 在哪台机器 |
|------------|------------|
| `主库检测` / `主库修复预览` / `主库一键修复` / `仅创建复制账号` / `重启MySQL` | **主库服务器（宝塔面板）** |
| `检测工具` / `GTID状态` / `主库连接测试向导` / `添加来源` / `保存映射` / `初始化任务` / `任务与来源日志` | **从库服务器（宝塔面板）** |
| 改主库 `my.cnf`、在主库执行建用户 SQL、放通主库 3306（云安全组/防火墙） | **主库服务器或云控制台** |
| 附录里的 `mysql -h ...` 命令行（可选） | **从库服务器**（仅高级排障时用） |

---

## 三、主线流程（按顺序做，不要乱序）

下面 1～7 是**唯一推荐顺序**。其中第 2 步在主库面板执行，其余在从库面板执行。

1. **体检**：`检测工具` → `GTID状态`（心里有数：只有 mysqldump 也能跑，大库再装物理工具）
2. **配主库（自动）**：切到主库服务器宝塔面板，执行 `主库检测` → `主库修复预览` → `主库一键修复`（可先不勾选「修复后自动重启」）；需要账号时在「主库自动修复增强」里填复制账号后，可「仅创建复制账号」或让一键修复一并处理  
3. **测连通**：`主库连接测试向导`（通过即可，**不需要**先跑命令行）
4. **加来源**：填写主库 IP / 端口 / 复制账号，点 `添加`
5. **写映射**：在表格里填「主库库名 → 从库目标库名」，点 `保存映射`（也可点 `载入已有映射` 编辑）
6. **初始化**：初始化任务选 `auto`，`创建任务` → 填任务 ID → `后台执行`
7. **验收**：`任务详情` / `任务日志` → 来源行 `状态` / `诊断`

若第 2 步自动修复失败：先去**附录 B 手工主库配置**，再回到第 3 步继续。

---

## 四、分步说明（与第三节序号一致）

### 第 1 步：体检（从库插件）

**位置：从库 → 宝塔 → 本插件**

1. 点 `检测工具`：看是否有 `mysqldump` / `mysql`；大库可再装物理工具（见下方「物理工具怎么选」）。
2. 点 `GTID状态`：确认从库侧能看到 GTID 相关信息（与多源复制要求一致）。

说明：只有 `mysqldump` 时，初始化模式选 `auto` 即可；物理工具用插件里「物理工具安装助手」装。

#### 物理工具怎么选（可选阅读）

- **xtrabackup**：MySQL 大库常用，物理热备快；依赖环境与版本匹配。
- **mariabackup**：MariaDB 环境优先。
- **mysqldump + mysql**：万能兜底，小库或先跑通流程最合适；大库会慢。

**一句话**：拿不准就选初始化模式 `auto`；库特别大再考虑装物理工具。

#### 「修复后自动重启」怎么用

- 新手建议：第一次**不勾选**自动重启，看清结果后，在低峰期再点 `重启MySQL`。
- 熟练后可勾选一键自动重启。

---

### 第 2 步：主库自动配置（在主库服务器执行）

**位置：主库服务器 → 宝塔 → 本插件**

顺序：

1. `主库检测`
2. `主库修复预览`
3. `主库一键修复`（按需勾选「修复后自动重启」）

页面「主库自动修复增强」里可填 `repl_user / repl_password / replica_host`（如 `%` 或从库 IP）：

- 需要单独建账号时，可点 **仅创建复制账号**
- 或在一键修复时一并创建（视你当前插件选项而定）

**成功参考**：检测报告里关键项多为 `ok`；修复返回 `snapshot_id`；若提示要重启，在低峰期 `重启MySQL`。

---

### 第 3 步：连通性（从库插件，不要先敲命令行）

**位置：从库 → 本插件**

点 `主库连接测试向导`，看到成功即可进入下一步。

- 默认**不要**用附录 A 的命令行；只有插件测不通、需要人工对照时才看附录 A。

---

### 第 4 步：添加主库来源

**位置：从库 → 本插件**

在「添加主库来源」示例：

| 项 | 示例 |
|----|------|
| 来源ID | `m1`（自定，短一点） |
| 通道名 | `ch_m1`（仅字母数字下划线） |
| 主库地址 | `10.0.0.11` |
| 端口 | `3306` |
| 复制账号/密码 | 与前面一致 |

点 `添加`。成功：列表中出现 `m1`，且无报错。

---

### 第 5 步：库映射（必须）

**位置：从库 → 本插件**

1. 「来源ID」填 `m1`（或点 `载入已有映射` 拉取已保存的项）
2. 在表格里「添加一行」，填写：
   - 主库库名：`order_db`
   - 从库目标库名：`m1_order_db`
3. 点 `保存映射`。成功：提示「映射已保存」。

（高级用户可展开「JSON 预览」核对；后端仍存储为 `source_db/target_db` 列表。）

---

### 第 6 步：初始化任务并后台执行

**位置：从库 → 本插件**

1. 来源 ID：`m1`，模式：`auto`（推荐）
2. `创建任务` → 记下任务 ID
3. 任务 ID 填入框内 → `后台执行`

---

### 第 7 步：看结果

**位置：从库 → 本插件**

按顺序：

1. `任务详情`：看 `status / progress / current_step`
2. `任务日志`：看失败原因
3. 来源行 `状态`：IO / SQL
4. 来源行 `诊断`：建议

**成功参考**：任务 `status=done`，通道 IO/SQL 正常，无持续报错。

---

## 五、两主一从示例（多主时重复第 4～7 步）

**位置：从库 → 本插件**

**主库 1**

- 来源 ID `m1`，通道 `ch_m1`
- 映射：表格一行 `order_db → m1_order_db`（等价 JSON：`[{"source_db":"order_db","target_db":"m1_order_db"}]`）

**主库 2**

- 来源 ID `m2`，通道 `ch_m2`
- 映射：表格一行 `crm_db → m2_crm_db`

注意：各来源的 **target_db 不要重名**。

## 六、报错排查顺序（全 UI）

**位置：从库 → 本插件**

1. 来源行 `连通性`
2. 来源行 `诊断`
3. `任务日志` → `来源日志`
4. `审计日志` → `回滚快照`（高危前确认）

---

## 七、常见报错速查

| 现象 | 优先处理 |
|------|----------|
| 请先配置库映射 | 先保存映射再建任务 |
| 主库网络不通 | 安全组/防火墙/路由/主库 3306 |
| access denied | 复制账号权限或主机限制（如 `user@host`） |
| GTID 异常 | 主库 `gtid_mode=ON` 等（优先用插件主库检测） |
| Unknown database | `source_db` 写错或主库无此库 |
| 任务一直 running/pending | 看任务日志；必要时取消重建或联系运维 |

## 附录 A：命令行连通性兜底（默认可跳过）

**何时用**：插件里 `主库连接测试向导` 失败，需要在对端用同一账号核对。  
**在哪执行**：从库服务器命令行。

```bash
mysql -h 主库IP -P3306 -urepl_user -p
```

---

## 附录 B：手工主库配置（仅自动修复失败时）

**位置：主库服务器（MySQL / 系统 / 云控制台）**

### B.1 检查 GTID / binlog

在主库 MySQL 执行：

```sql
SHOW VARIABLES LIKE 'gtid_mode';
SHOW VARIABLES LIKE 'enforce_gtid_consistency';
SHOW VARIABLES LIKE 'log_bin';
SHOW VARIABLES LIKE 'binlog_format';
SHOW VARIABLES LIKE 'server_id';
```

不理想则改主库 `my.cnf` 的 `[mysqld]` 后重启，例如：

```ini
gtid_mode=ON
enforce_gtid_consistency=ON
log_bin=ON
binlog_format=ROW
server_id=101
```

### B.2 创建复制账号

把 `10.0.0.2` 换成从库 IP：

```sql
CREATE USER 'repl_user'@'10.0.0.2' IDENTIFIED BY 'Repl@2026!';
GRANT REPLICATION SLAVE, REPLICATION CLIENT ON *.* TO 'repl_user'@'10.0.0.2';
FLUSH PRIVILEGES;
```

MySQL 8.0 也可用：

```sql
GRANT REPLICATION REPLICA, REPLICATION CLIENT ON *.* TO 'repl_user'@'10.0.0.2';
```

### B.3 放行 3306

- 云安全组：从库 IP → 主库 3306  
- 主机防火墙：同上  

做完附录 B 后，回到**第三节主线**：在从库服务器从第 3 步「测连通」继续。
