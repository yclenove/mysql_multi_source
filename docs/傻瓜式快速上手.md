# 傻瓜式快速上手（小白专用）

这份文档目标只有一个：  
**让你不懂主从原理，也能把插件配置成功并跑起来。**

---

## 一眼看懂：你只要准备 6 个信息

先把下面 6 项写在记事本里，后面会反复用到：

1. 从库IP（例如 `10.0.0.2`）
2. 主库IP（例如 `10.0.0.11`）
3. 主库端口（一般 `3306`）
4. 复制账号（例如 `repl_user`）
5. 复制密码（例如 `Repl@2026!`）
6. 主库库名与从库目标库名（例如 `order_db -> m1_order_db`）

---

## 第0步：环境要求（不满足就先别往下）

- 从库：宝塔 + MySQL 已安装
- 主库：MySQL 可正常登录
- 主库到从库网络可达，3306 放行
- 插件已安装并能打开

---

## 第1步：主库配置（每台主库都要做）

### 1.1 检查 GTID 和 binlog

在主库执行：

```sql
SHOW VARIABLES LIKE 'gtid_mode';
SHOW VARIABLES LIKE 'enforce_gtid_consistency';
SHOW VARIABLES LIKE 'log_bin';
SHOW VARIABLES LIKE 'binlog_format';
SHOW VARIABLES LIKE 'server_id';
```

理想结果：
- `gtid_mode=ON`
- `enforce_gtid_consistency=ON`
- `log_bin=ON`
- `binlog_format=ROW`
- `server_id` 是非 0 唯一值

如果不是，编辑主库 `my.cnf` 的 `[mysqld]`：

```ini
gtid_mode=ON
enforce_gtid_consistency=ON
log_bin=ON
binlog_format=ROW
server_id=101
```

重启 MySQL 后再查一次。

### 1.2 创建复制账号

把 `10.0.0.2` 换成从库IP：

```sql
CREATE USER 'repl_user'@'10.0.0.2' IDENTIFIED BY 'Repl@2026!';
GRANT REPLICATION SLAVE, REPLICATION CLIENT ON *.* TO 'repl_user'@'10.0.0.2';
FLUSH PRIVILEGES;
```

MySQL 8.0 也可用：

```sql
GRANT REPLICATION REPLICA, REPLICATION CLIENT ON *.* TO 'repl_user'@'10.0.0.2';
```

### 1.3 放行 3306（别漏）

- 云安全组：放行从库IP -> 主库3306
- 主机防火墙：放行从库IP -> 主库3306

### 1.4 连通性验收（从库执行）

```bash
mysql -h 主库IP -P3306 -urepl_user -p
```

能连上，才进入插件配置。

---

## 第2步：打开插件先做体检

打开插件后先点：

1. `检测工具`
2. `GTID状态`

说明：
- 你只有 `mysqldump` 也能跑，模式选 `auto` 即可
- 若需要物理工具，可用插件里的“物理工具安装助手”

### 2.1 主库修复时“自动重启”是可选的

页面上有：
- `主库一键修复`
- `修复后自动重启MySQL`（复选框）

建议新手做法：
1. 第一次先**不勾选**自动重启，先看修复结果
2. 修复成功后，在业务低峰再手工点 `重启MySQL`
3. 熟悉后可勾选“修复后自动重启”实现一步到位

---

## 第3步：添加主库来源（照填）

在“添加主库来源”填：

- 来源ID：`m1`（你自己定，短一点）
- 通道名：`ch_m1`（只用字母数字下划线）
- 主库地址：`10.0.0.11`
- 主库端口：`3306`
- 复制账号：`repl_user`
- 复制密码：`Repl@2026!`

点 `添加`。

### 成功标志
- 不报错
- 来源列表出现一条 `m1`

---

## 第4步：保存库映射（必须做）

“库映射配置（JSON）”输入：

```json
[
  {"source_db":"order_db","target_db":"m1_order_db"}
]
```

来源ID填 `m1`，点 `保存映射`。

### 成功标志
- 弹“映射已保存”

---

## 第5步：创建初始化任务并后台执行

1. 初始化任务区域：
   - 来源ID：`m1`
   - 模式：`auto`（推荐）
2. 点 `创建任务`
3. 复制返回的任务ID（例如 `boot_xxxxx`）
4. 粘贴到“任务ID”输入框
5. 点 `后台执行`

---

## 第6步：看结果（重点）

按顺序看：

1. `任务详情`：看 `status/progress/current_step`
2. `任务日志`：看失败点
3. 来源行点 `状态`：看 IO/SQL 状态
4. 来源行点 `诊断`：看建议

### 成功标志
- 任务 `status=done`
- 来源状态 IO/SQL 为运行
- 没有持续报错

---

## 第7步：两主一从示例（可直接套）

### 主库1
- 来源ID：`m1`
- 通道：`ch_m1`
- 映射：
```json
[{"source_db":"order_db","target_db":"m1_order_db"}]
```

### 主库2
- 来源ID：`m2`
- 通道：`ch_m2`
- 映射：
```json
[{"source_db":"crm_db","target_db":"m2_crm_db"}]
```

注意：目标库名不要重复，避免冲突。

---

## 第8步：报错就按这个顺序排查（别乱试）

1. 来源行点 `连通性`
2. 来源行点 `诊断`
3. 点 `任务日志`
4. 点 `来源日志`
5. 再看主库 MySQL 错误日志

---

## 最常见报错 -> 直接处理

- “请先配置库映射”
  - 先保存 mapping 再创建任务

- “主库网络不通”
  - 检查主库3306、安全组、防火墙、路由

- “access denied / 权限不足”
  - 复制账号权限不够或IP限制不对

- “GTID 异常”
  - 主从先确认 `gtid_mode=ON`

- “Unknown database”
  - `source_db` 写错，或者主库不存在该库

- “任务一直 running/pending”
  - 看任务日志，必要时调用恢复接口或取消重建

---

## 上线前最后检查（建议照抄）

- [ ] 先拿一个小库跑通全流程
- [ ] 两个来源都能单独跑通
- [ ] 来源诊断无严重报错
- [ ] 任务失败能看到明确日志
- [ ] 确认目标库命名无冲突

做到这一步，小白也能稳定配置成功。
