# 设计文档

## 1. 背景与目标

当前项目目标是实现宝塔插件 `mysql_multi_source`，解决“单从库实例接入多个主库”的多源复制场景，并兼顾大库初始化效率与运维可用性。

## 2. 设计原则

- 单一职责：配置管理、复制控制、任务调度、状态监控分层设计
- 低耦合：前端与后端通过稳定 API 协作，后端模块边界清晰
- 可扩展：支持后续新增复制模式、初始化策略、告警通道
- 可观测：每个 source/channel 的状态、日志、错误单独可追踪
- 可回滚：高风险操作需确认，流程可中断并恢复

## 3. 总体架构

- 前端：`index.html`（后续拆分多页面模块）
- 后端：`mysql_multi_source_main.py`
- 后台任务入口：`start_sync.py`
- 配置存储：`multi_source_info.json`
- 日志目录：`log/`（按 source/channel 切分）

## 4. 领域模型

### 4.1 从库实例（slave_instance）
- `host`
- `port`
- `version`
- `updated_at`

### 4.2 数据源（source）
- `source_id`
- `channel_name`
- `master_host`
- `master_port`
- `repl_user`
- `repl_password`（后续加密）
- `sync_mode`（默认 GTID）
- `db_mappings`（主库库名 -> 从库库名）
- `status`（running/io/sql/seconds_behind/last_error）

## 5. 核心流程

1. 新增 source
2. 校验参数与连接
3. 配置 channel
4. 启动 channel
5. 周期获取状态与错误
6. 用户按 source/channel 进行启停与诊断

## 6. 性能设计（规划）

- 大库初始化优先走物理方式（xtrabackup 等）
- 逻辑方式作为兜底（并行导入）
- 分阶段任务与进度反馈，避免用户无感等待

## 7. 风险与约束

- 不允许无 channel 的全局复制操作
- 不允许明文输出敏感信息
- 删除/重置类操作必须二次确认

## 8. 后续演进

- 增加分 channel 诊断报告
- 增加初始化策略编排器
- 增加告警与自动修复能力

## 9. 已落地能力（阶段 3-5）

- 映射层：支持 source 级 `source_db -> target_db` 映射维护
- 编排层：支持初始化任务创建、执行编排、取消、查询
- 诊断层：支持来源诊断建议、来源日志、系统总览指标
- 安全层：密码脱敏显示、关键动作日志留痕
