# 设计文档

## 1. 背景与目标

当前项目目标是实现宝塔插件 `mysql_multi_source`，解决“单从库实例接入多个主库”的多源复制场景，并兼顾大库初始化效率与运维可用性。

## 2. 设计原则

- 单一职责：配置管理、复制控制、任务调度、状态监控分层设计
- 低耦合：前端与后端通过稳定 API 协作，后端模块边界清晰
- 可扩展：支持后续新增复制模式、初始化策略、告警通道
- 可观测：每个 source/channel 的状态、日志、错误单独可追踪
- 可回滚：高风险操作需确认，流程可中断并恢复

## 3. 总体架构

- 前端：`index.html`（后续拆分多页面模块）
- 后端：`mysql_multi_source_main.py`
- 后台任务入口：`start_sync.py`
- 配置存储：`multi_source_info.json`
- 日志目录：`log/`（按 source/channel 切分）

## 4. 领域模型

### 4.1 从库实例（slave_instance）
- `host`
- `port`
- `version`
- `updated_at`

### 4.2 数据源（source）
- `source_id`
- `channel_name`
- `master_host`
- `master_port`
- `repl_user`
- `repl_password`（后续加密）
- `sync_mode`（默认 GTID）
- `db_mappings`（主库库名 -> 从库库名）
- `status`（running/io/sql/seconds_behind/last_error）

## 5. 核心流程

1. 新增 source
2. 校验参数与连接
3. 配置 channel
4. 启动 channel
5. 周期获取状态与错误
6. 用户按 source/channel 进行启停与诊断

## 6. 性能设计（规划）

- 大库初始化优先走物理方式（xtrabackup 等）
- 逻辑方式作为兜底（并行导入）
- 分阶段任务与进度反馈，避免用户无感等待

## 7. 风险与约束

- 不允许无 channel 的全局复制操作
- 不允许明文输出敏感信息
- 删除/重置类操作必须二次确认
- 多源映射不得静默写入同名 `target_db`，必须在接入前显式预检并阻断
- 重复接入属于幂等更新流程，失败时只能恢复原配置，不能删除既有来源

## 8. 后续演进

- 增加分 channel 诊断报告
- 增加初始化策略编排器
- 增加告警与自动修复能力

## 9. 已落地能力（阶段 3-5）

- 映射层：支持 source 级 `source_db -> target_db` 映射维护
- 编排层：支持初始化任务创建、执行编排、取消、查询
- 诊断层：支持来源诊断建议、来源日志、系统总览指标
- 安全层：密码脱敏显示、关键动作日志留痕

## 10. 已落地能力（阶段 7-11）

- 双引擎执行层：physical/logical 两种初始化路径
- 可靠任务模型：心跳、重试、checkpoint、worker并发保护
- 运维强化：任务日志、恢复接口、压测脚本与运维文档

## 11. 已落地能力（阶段 14）

- 主库协同模式：角色切换、健康检测、自动修复、重启联动
- 双通道交互：签名配置文件导入 + API握手会话
- 安全治理：审计日志、变更快照、快照回滚

## 12. 已落地能力（阶段 18）

- 前端交互架构增强（仍保持单文件无构建链）：
  - 新增请求适配层（错误标准化、singleFlight、防旧请求覆盖）
  - 新增统一结果面板与统一弹窗出口，减少信息丢失与拼接渲染风险
  - 新增按钮执行态封装，降低重复触发导致的并发副作用
- 安全增强：
  - 来源列表渲染改为转义输出，修复关键 XSS 风险点
- 后端契约收敛：
  - 引入 `_ok/_fail` 辅助返回，关键接口补齐 `message/code`
  - `source_id` 规则校验、`master_port` 范围校验前置

## 13. 已落地能力（阶段 22）

- 接入护栏增强：
  - 新增 `check_target_db_conflicts`，用于检测“请求内重复映射”与“占用已有目标库”两类冲突
  - 从库向导在选库阶段和最终提交前都会触发该预检
- 回滚安全增强：
  - `wizard_start_replication` 在重复接入失败时改为恢复原来源快照
  - 避免 `set_db_mappings` 失败时误删既有来源、任务与通道配置

## 14. 已落地能力（阶段 27）

- 配置单安全增强：
  - `replica_verify_profile` 改为对 `signature` 执行真实验签，不再只做字段与过期时间检查
  - 配置单导入与握手接收链路统一复用该校验结果，避免伪造配置单绕过导入
- 向导流程一致性增强：
  - 从库向导在进入策略页前会等待目标库冲突预检完成
  - 发现冲突时直接留在当前步骤，避免“已进入下一步但稍后又提示冲突”的状态错位

## 15. 已落地能力（阶段 28）

- 握手可观测性增强：
  - 握手 session 持久化记录 `accept_attempts`、`last_error`、`last_error_code`、`last_error_at`、`accepted_at`
  - 握手失败时审计日志会记录失败原因与错误码，便于回溯签名失败或导入失败
- 总览状态聚合收敛：
  - `overview_metrics()` 改为复用 `_all_slave_status()` 与 `_map_status_row()` 一次抓取全部 channel 状态
  - 与 `wizard_dashboard_snapshot()` 共享同一套状态映射逻辑，避免逐来源 N+1 查询
- 重复触发护栏增强：
  - 已取消的 bootstrap 任务禁止直接重新触发
  - 从库向导提交阶段增加独立执行锁，避免重复点击导致的并发提交
