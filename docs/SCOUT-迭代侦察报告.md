## 二、市场竞品分析

### 2.1 竞品矩阵

| 维度 | mysql_multi_source | ProxySQL | MaxScale | Orchestrator | MySQL Shell |
|------|-------------------|----------|----------|--------------|-------------|
| 定位 | 宝塔多源复制 GUI | 读写分离代理 | 数据库代理/路由 | 复制拓扑管理 | MySQL 官方 CLI |
| 多源复制 GUI | 专精 | 不支持 | 不支持 | 仅可视化 | 无 GUI |
| 宝塔集成 | 原生 | 无 | 无 | 无 | 无 |
| 初始化编排 | 物理+逻辑双引擎 | 无 | 无 | 无 | 有但 CLI |
| 向导引导 | 5 步从库向导 | 无 | 无 | 无 | 无 |
| 握手/签名配置 | 支持 | 无 | 无 | 无 | 无 |
| 学习成本 | 低（GUI 向导） | 高（SQL 配置） | 中（配置文件） | 中 | 高（CLI） |
| 目标用户 | 中小站长/DBA | 企业 DBA | 企业 DBA | DBA | DBA |

### 2.2 差异化优势

1. **唯一性**：宝塔插件生态中无任何多源复制管理工具，本项目是该细分赛道唯一产品
2. **向导编排**：5 步从库接入向导 + 4 步主库运维向导，竞品均无此体验
3. **双引擎初始化**：物理（xtrabackup）+ 逻辑（mysqldump）自动降级，覆盖更多场景
4. **握手会话**：签名配置文件 + token 握手，安全地跨机器导入配置，竞品无此设计
5. **零门槛**：宝塔用户无需学习新工具，插件即装即用

## 三、用户痛点清单

### 3.1 目标用户画像

- **主要用户**：使用宝塔面板管理 MySQL 的中小站长、运维人员
- **次要用户**：中小企业的初级 DBA
- **典型场景**：多业务库汇聚到一个从库做报表/分析/备份

### 3.2 痛点矩阵

| 编号 | 痛点 | 严重度 | 当前覆盖 | 说明 |
|------|------|--------|----------|------|
| P-1 | 宝塔无多源复制 GUI，只能手动敲 SQL | 高 | 已解决 | 向导 + 仪表盘完全覆盖 |
| P-2 | 大库初始化慢且容易中断 | 高 | 已解决 | 物理/逻辑双引擎 + 断点恢复 |
| P-3 | 多 channel 状态监控困难 | 高 | 已解决 | 仪表盘卡片 + 健康灯 + 5s 轮询 |
| P-4 | 配置出错后难以回滚 | 中 | 已解决 | 快照回滚 + 冲突预检 |
| P-5 | 主库配置不熟悉，不知道要开什么 | 中 | 已解决 | 主库协同向导 + 一键修复 |
| P-6 | 复制延迟告警缺失 | 中 | 未覆盖 | 无告警通知机制（邮件/webhook） |
| P-7 | 备份与多源复制冲突 | 中 | 未覆盖 | 宝塔备份恢复后 channel 配置可能丢失 |
| P-8 | 团队协作时配置同步困难 | 低 | 部分覆盖 | 签名配置文件可导出，但无版本管理 |
| P-9 | 无操作审计追溯 | 低 | 已解决 | 审计日志 + 握手留痕 |

### 3.3 用户旅程关键摩擦点

```
安装 -> 首次使用 -> 接入第一个主库 -> 初始化数据 -> 持续监控
  |         |              |                |              |
  v         v              v                v              v
 零摩擦   向导引导好     连通性检查到位   双引擎自动降级   缺告警通知
```

最大摩擦点：**持续监控阶段缺乏主动告警**，用户必须主动打开面板才能发现问题。
